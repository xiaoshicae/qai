# 代码审查策略（借鉴 Cursor Bugbot）

> 参考：[Building Bugbot](https://cursor.com/blog/building-bugbot)

## 核心原则

### 1. 边界情况优先

对每一处变更，首要问题是：**这段代码在什么条件下会出问题？**

重点检查：
- 空值 / 空数组 / 空对象
- 零值 / 负值 / 极大值
- 并发 / 竞态条件（如 Mutex 锁、连续触发、重复请求）
- 网络失败 / 超时 / 部分成功
- 上游返回非预期格式

### 2. 追踪副作用链

不只看当前代码块，追踪变更的**下游影响**：
- 返回值被谁消费？消费方是否处理了所有可能的返回值？
- 错误是否被静默吞掉？是否导致后续步骤在错误状态下继续执行？
- 状态变更后，依赖该状态的其他逻辑是否还能正确工作？

### 3. 动态上下文拉取

不要只看 diff，主动拉取**周围代码**理解完整语义：
- 被调用函数的签名和返回值
- 被修改变量的其他使用位置
- 配置项的实际消费方
- 类似模式在项目中的其他实现（是否有不一致）

### 4. 攻击性探测

对每个可疑模式都**深入调查**，宁可多查不漏：
- 看到字符串比较 → 检查是否有 `null` / `undefined` / `"null"` 字符串混淆
- 看到条件分支 → 检查是否覆盖所有可能路径
- 看到错误处理 → 检查是否真的阻止了后续执行
- 看到 `unwrap()` → 检查是否可能 panic
- 看到 SQL 拼接 → 检查是否参数化

### 5. 关注真 bug，忽略噪音

**必须报告**（P0）：
- 逻辑错误（条件反转、遗漏分支）
- 边界情况导致的运行时错误 / panic
- 安全漏洞（注入、凭证泄露、权限绕过）
- 静默失败（错误被吞掉、虚假成功状态）
- 数据丢失 / 数据不一致

**不要报告**：
- 纯风格问题（已有 lint 检查）
- 文档措辞
- 命名偏好（除非造成歧义）

## 审查检查清单

对每个变更的函数 / 代码块，逐条过：

```
[ ] 输入为空/null/undefined 时行为是否正确？
[ ] 错误路径是否会阻止后续执行（而非静默继续）？
[ ] 返回值的所有可能形态是否都被消费方处理？
[ ] 外部调用失败时是否有正确的错误传播？
[ ] 并发/重复执行时是否安全？
[ ] 状态变更是否会影响其他依赖方？
```

## 在各流程中的应用

| 流程 | 应用方式 |
|------|---------|
| `/commit` | 提交前快速扫描：边界情况 + 静默失败 |
| `/push` | 规范合规审查中增加边界情况检查 |
| `/code-review` | 作为审查方法论的核心，逐文件应用 |
| `/pr` | PR 描述中标注已识别的边界情况和处理方式 |
| `/ship` | 整合到质量审查阶段 |
